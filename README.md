# key_pipeline

`key_pipeline` contains the structured, pipeline-oriented portion of the project. It separates active workflow code from experiment-era datasets and keeps the processing flow, data layout, and historical artifacts explicit.

At this stage, the scope covers article ingestion, intermediate dataset preparation, metadata recovery, prediction, and lightweight evaluation. Macro merge, intraday alignment, and DXY-based evaluation remain outside the active path.

## Scope

`key_pipeline` covers the following workflow:

1. load or ingest article data
2. normalize and clean intermediate datasets
3. recover publication metadata and full text where available
4. run prediction or classification
5. write outputs in a consistent structure
6. preserve experiment artifacts separately from the active pipeline

## Directory responsibilities

### `docs/`

Project documentation for pipeline flow, experiment definitions, data organization, and restructuring rules.

* `pipeline_overview.md` documents the active processing flow
* `experiment_map.md` defines the meaning of experiment designs 1, 2, and 3
* `data_layout.md` documents active, experimental, output, and archive data boundaries
* `reorg_rules.md` records naming and movement conventions used during cleanup

### `pipeline/`

Active, script-first workflow code.

* `ingest/` contains data loading and article ingestion logic
* `extract/` contains publication-time recovery, metadata recovery, and full-text preparation
* `predict/` contains model or prompt-based inference
* `evaluate/` contains evaluation and result summarization
* `run_pipeline.py` is the canonical pipeline entrypoint

### `data/active/`

Datasets read by the active code path.

### `data/experimental/`

Historical experiment-era datasets grouped by experiment design.

### `data/archive/`

Historical or deprecated files retained for traceability but excluded from the active pipeline.

### `data/outputs/`

Outputs generated by the active workflow.

### `scripts/`

Utility scripts that do not define core pipeline stages.

## Pipeline shape

The active flow inside `key_pipeline` follows this sequence:

1. ingest or load article input
2. clean and standardize intermediate tables
3. recover publication fields and full text where available
4. assemble model-ready input
5. run prediction
6. save outputs
7. run lightweight evaluation

This keeps the active path narrow and reproducible while preserving older experiments in parallel.

## Experimental data

`data/experimental/` contains the development history of the project in three grouped experiment designs. Each experiment is split into:

* `input/` — datasets prepared before or during model execution
* `result/` — model outputs, filled outputs, mapped outputs, or post-processed results

This layout preserves chronology and makes it possible to trace later outputs back to their input tables.

## Special file

### `data/experimental/us.py`

`us.py` lives under `data/experimental/` and remains outside the CSV manifest. It is tracked separately from the experiment tables.

## Naming conventions

The experimental filenames are historical and abbreviated. The abbreviations carry stage information:

* `cln` = cleaned
* `ft` = full text
* `pub` = publication-related fields or publish-time processing
* `mapped` = mapped or aligned dataset
* `w_macro` = with macro features
* `res` = result output
* `filled` = backfilled or completed output
* `timeout` = timeout or incomplete result
* `cutoff` = filtered by experiment cutoff rule

These names are stage markers rather than full semantic descriptions.

## Path conventions

The experimental directories use the live structure:

```text
key_pipeline/data/experimental/<experiment_id>/input/<file>.csv
key_pipeline/data/experimental/<experiment_id>/result/<file>.csv
```

Some manifest entries still reflect an older flat form without `input/` or `result/`. The active directory layout is the source of truth.

## Operating principles

The structure follows a small set of rules:

* active files live outside `archive/`
* experiment-specific files stay inside their experiment folders
* historical but retained files live in `archive/`
* each experiment preserves the `input/` and `result/` split
* path updates remain aligned with code references
* naming reflects pipeline stage first and historical interpretation second

## Active pipeline boundary

The current active boundary for `key_pipeline` is:

* article input
* cleaning and collation
* publication metadata recovery
* full-text attachment where available
* model-ready input assembly
* prediction
* output writing
* lightweight evaluation

Macro features, intraday market alignment, and DXY-based evaluation remain outside the active pipeline boundary.
